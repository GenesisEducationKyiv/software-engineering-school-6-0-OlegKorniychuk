# ADR-002: Object-Relational Mapping (ORM) Selection

Status: Accepted \
Date: 09.05.2026 \
Author: Oleh Korniichuk

## Context

Our API needs to interact with a PostgreSQL database to manage user subscriptions and repository tracking data. We need a reliable, type-safe, and performant way to construct queries, manage schema migrations, and map database rows to TypeScript objects without introducing unnecessary overhead.

## Decision

Use Drizzle ORM

## Alternatives Considered

1. TypeORM
   - Upsides: highly mature, massive ecosystem, supports both Active Record and Data Mapper patterns, integrates seamlessly with enterprise frameworks
   - Downsides: heavy bundle size, relies heavily on decorators and reflection (which can impact performance), often generates unpredictable or inefficient SQL for complex relations
2. Prisma
   - Upsides: best-in-class developer experience, highly readable custom schema file, excellent autocomplete
   - Downsides: uses a heavy Rust-based query engine that consumes significant memory, abstracts SQL too much, not natively edge-compatible
3. Drizzle ORM
   - Upsides: extremely lightweight, strictly typed, edge-compatible, generates raw and predictable SQL, allows writing SQL-like syntax directly in TypeScript, highly performant
   - Downsides: newer ecosystem with fewer community resources, requires a strong understanding of raw SQL, lacks the extensive built-in abstraction for deep nested relations

## Consequences

- **Positive**: We achieve maximum database performance with minimal overhead and boilerplate. We retain full control over our SQL queries while still enjoying strict TypeScript safety, meaning fewer runtime errors and faster execution
- **Negative**: The team must write slightly more verbose query logic. We are also adopting a relatively new tool, meaning we may encounter edge cases and bugs
