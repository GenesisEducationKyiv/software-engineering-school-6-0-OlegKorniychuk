# ADR-003: Restructure to Modular Monolith

Status: Accepted \
Date: 06.06.2026 \
Author: Oleh Korniichuk

## Context

The current codebase organizes code by technical role: `src/services/`, `src/repositories/`, `src/modules/` (only one module exists). This flat, layer-first structure makes it hard to reason about business boundaries вЂ” a single feature change touches four unrelated directories. There is also no explicit public API between domains, meaning any service can reach into any repository directly. This tight coupling makes future extraction into microservices expensive.

## Decision

Restructure to a modular monolith with three domain modules and a shared kernel.

### Modules

**`subscription`** вЂ” user-facing subscription lifecycle. Owns the `subscriptions` table, subscription repository, business logic (subscribe / confirm / unsubscribe / list), notification token generation, and HTTP controllers. This is the system's core domain.

**`tracker`** вЂ” GitHub integration and release detection. Owns the `github_repositories` table, GitHub API client, repository scanner, release checker, and the hourly scanner cron. Responsible for detecting new releases and triggering notifications.

**`notification`** вЂ” outbound email delivery. Owns the BullMQ email queue and worker, Nodemailer client, and email templates. Receives work via its public API; knows nothing about subscriptions or GitHub.

### Shared kernel (`src/shared/`)

Infrastructure used by all modules with no business logic: DB client, Redis connection, auth middleware, cache service, error handling, config, logger, metrics.

### Module communication rules

- Each module exposes a facade interface as its only public API. Other modules import the facade, never internal repositories or services.
- `subscription` calls `tracker` facade to verify a GitHub repository exists on subscribe.
- `tracker` calls `notification` facade to dispatch emails when a new release is detected. The subscriber list is fetched by `tracker` via the `subscription` facade (not by reaching into the subscription repository directly).
- Within a module, internal files may import each other freely.

## Alternatives Considered

1. **Keep current layer-first structure** вЂ” No upfront cost, but cross-domain coupling grows with every feature. Splitting into microservices later would require untangling the entire codebase at once.

2. **Go directly to microservices** вЂ” Premature. The domain boundaries are not yet proven under real load or real product requirements. Distributed systems add operational overhead (service discovery, inter-service auth, distributed tracing) that is not justified at this stage.

3. **Modular monolith** _(chosen)_ вЂ” Enforces domain boundaries and explicit public APIs inside a single deployable. Operational complexity stays low. When a module needs to scale or be owned by a separate team, extraction is mechanical: replace the in-process facade call with an HTTP/gRPC/queue call.

## Consequences

- **Positive**: Each domain is self-contained and can be understood, tested, and changed in isolation. Cross-boundary dependencies are explicit and auditable. Microservice extraction becomes a matter of replacing facade calls with network calls, not restructuring the entire codebase.
- **Negative**: More upfront reorganization effort. Developers must respect the module boundary rules (linting rules or code review); the compiler alone does not enforce them.
