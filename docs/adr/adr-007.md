# ADR-007: HTTP vs gRPC for Notification Service Transport

## Status
Accepted

## Context

The `api` service communicates with the `notification` microservice to queue confirmation emails and dispatch release notifications. This was originally implemented as REST over HTTP/1.1 using `HttpNotificationFacade`. As the system evolved toward a microservices architecture, we evaluated replacing HTTP with gRPC (via ConnectRPC + protobuf-es) to understand the trade-offs.

Both transports are retained and switchable via `NOTIFICATION_TRANSPORT=http|grpc` (default: `grpc`).

## Decision Drivers

- Developer experience: proto file as single source of truth for the service contract
- Type safety: generated TypeScript types shared between server and client
- Performance: throughput and latency for inter-service calls
- Operational simplicity: ease of debugging, tooling

## Options Considered

### Option A: HTTP REST (existing)
Plain JSON over HTTP/1.1. Implemented via `HttpNotificationFacade` calling `/emails/confirmation` and `/emails/dispatch`.

### Option B: gRPC via ConnectRPC + protobuf-es
- `proto/notification/v1/notification.proto` defines the service contract
- `buf generate` produces TypeScript types from the proto (single source of truth)
- `@connectrpc/connect-node` provides client (`createGrpcTransport`) and server (`connectNodeAdapter` + `http2.createServer`)
- Service implementation: `NotificationGrpcHandler`
- Client: `GrpcNotificationFacade`

## Decision

**gRPC (ConnectRPC) is the default transport.** HTTP is kept as a fallback via env var.

Rationale:
- Proto file is the canonical contract — no drift between client and server types
- Generated TypeScript types eliminate manual interface maintenance
- `buf lint` and `buf breaking` guard against incompatible proto changes
- gRPC is the conventional choice for synchronous inter-service calls in microservice architectures

## Benchmark Results

**Setup:** Local machine, Redis via Docker. Notification service started with `tsx src/notification-server.ts`. Duration: 10s, Connections: 10.

**Methodology:**
- HTTP: `autocannon` against `POST /emails/confirmation` (HTTP/1.1, JSON body)
- gRPC: Custom concurrent loop using the generated gRPC client (`createGrpcTransport`) — batches of 10 concurrent calls, measured over the same 10s window. autocannon does not support h2c (HTTP/2 cleartext), so a direct comparison via the same tool is not possible.

| Metric          | HTTP REST  | gRPC ConnectRPC |
|-----------------|------------|-----------------|
| Requests/sec    | 2,574      | 635             |
| Latency p50     | 3 ms       | 13 ms           |
| Latency p99     | 10 ms      | 28 ms           |
| Errors          | 0          | 0               |
| Throughput      | ~380 KB/s  | N/A             |

**Interpretation:**

The raw numbers favor HTTP by ~4×, but the methodologies differ significantly:

- autocannon pipelines HTTP/1.1 requests over persistent connections — a best-case scenario for HTTP/1.1 REST
- The gRPC loop uses sequential batching (wait for all 10 to finish, then send next 10) which introduces artificial head-of-line blocking
- gRPC's HTTP/2 transport has per-request framing overhead and TLS/h2c setup cost that doesn't apply to the local HTTP/1.1 test

In practice, the benchmark does not isolate protocol overhead from benchmark methodology differences. For a fairer comparison, tools like `ghz` (gRPC-native load tester) should be used against the ConnectRPC server.

## Trade-offs

| | HTTP REST | gRPC (ConnectRPC) |
|---|---|---|
| Contract source of truth | Interface file (manual) | `.proto` file (generated) |
| Type safety | Manual | Generated |
| Tooling | curl, Postman | grpcurl, buf |
| Raw throughput (local) | Higher (autocannon) | Lower (methodology gap) |
| Protocol | HTTP/1.1 | HTTP/2 |
| Browser support | Yes | Via Connect protocol |
| Debugging | Easy (plain JSON) | Requires grpcurl / reflection |

## Consequences

- `proto/notification/v1/notification.proto` is the single source of truth for the notification service API
- `npm run proto:gen` must be re-run after any proto changes
- Generated files in `src/generated/` are committed to the repo
- The notification service exposes two servers: HTTP/1.1 on port 3002 (health + REST fallback) and HTTP/2 gRPC on port 50051
- `NOTIFICATION_TRANSPORT=http` can be set to revert to REST for debugging or if gRPC is unavailable
