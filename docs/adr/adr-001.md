# ADR-001: Mass Email Notification

Status: Accepted \
Date: 12.04.2026 \
Author: Oleh Korniichuk

## Context

After all repositories have been scanned a lot of emails must be sent at once. We need to work out a way of doing this without bloking the event loop.

## Decision

Use bullMQ

## Alternatives Considered

1. Promise.all
   - Upsides: the easiest to implement, zero external dependencies & infrastracture overhead
   - Downsides: unfit for scaling, may be blocked due to email provider API rate limit, risk socker exhaustion
2. In-memory async queue
   - Upsides: easy to implement, zero external dependencies & infrastructure overhead, no network overhead
   - Downsides: memory spike, impossible to distribute, no persistance if app crashes
3. BullMQ
   - Upsides: persists through crashes, easy to limit, easily scalable, automatic retries, utilizes Redis we already use for caching
   - Downsides: higher overhead for small workloads, makes Redis a hard dependency

## Consequences

- **Positive**: Our API will remain responsive even during high-traffic release windows. We gain the ability to scale "Email Notifier" workers independently from the "Subscription Service."

- **Negative**: We are now forced to manage and monitor a Redis instance. We also need to handle "Stuck" jobs that could occasionally occur in distributed queues.
